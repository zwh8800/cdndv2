# 上下文压缩改进方案

## 问题诊断

当前 `context_compressor.go` 存在以下核心缺陷：

1. **角色标识错误** — 压缩摘要输出为 `NewUserMessage`（L192-194），LLM 会将历史摘要误解为玩家输入
2. **检查点不足** — 仅在 `observe` 阶段检查一次（react_loop.go L295），但 `act` 阶段可能在一轮 ReAct 中追加大量 tool_call/result 消息导致超限
3. **保留策略粗糙** — 固定保留最近 10 条消息（RecentKeepCount=10），一次含 3 个 tool call 的操作就产生 7 条消息，10 条仅覆盖约 1.5 轮迭代
4. **Token 估算无反馈** — 纯启发式估算（~30-40% 误差），未利用 API 返回的实际 `Usage.PromptTokens` 校准
5. **无 D&D 领域感知** — 不区分查询类工具（数据会在 system prompt 中重新注入）和动作类工具（结果不可恢复），对所有工具结果同等截断到 100 字符
6. **压缩阻塞主流程** — 当前 `maybeCompressHistory()` 是同步调用，压缩过程（尤其是未来引入 LLM 摘要后）会阻塞用户的 ReAct 循环，增加响应延迟

## 改进内容

### Task 1: 重写 ContextCompressor 结构体与配置

**文件**: `game_engine/llm/context_compressor.go`

改造 `ContextCompressor` 结构体，增加以下字段：

```go
type ContextCompressor struct {
    ContextWindowSize int
    CompressThreshold float64
    RecentKeepRounds  int     // 改为按"对话轮次"保留（默认 3），替代 RecentKeepCount
    
    // Token 估算校准
    lastActualTokens   int     // 上次 API 返回的实际 prompt token 数
    lastEstimatedTokens int    // 上次估算值
    calibrationRatio   float64 // 校准系数，默认 1.3（偏保守）
}
```

- `RecentKeepCount` 改名为 `RecentKeepRounds`，语义从"消息条数"变为"完整对话轮次数"
- 新增 `calibrationRatio` 字段，初始 1.3（偏保守），后续通过实际 token 反馈动态调整
- `DefaultContextCompressor()` 默认 `RecentKeepRounds: 3, calibrationRatio: 1.3`

### Task 2: 修复压缩输出的消息角色

**文件**: `game_engine/llm/context_compressor.go` — `compressOldMessages()` 方法

将压缩摘要的输出从 `NewUserMessage` 改为 `NewSystemMessage`：

```go
// 修改前：
return []Message{NewUserMessage(summaryContent)}

// 修改后：
return []Message{NewSystemMessage(summaryContent)}
```

system message 不会被 LLM 误解为玩家输入，且语义上更合适（这是系统提供的背景上下文）。

### Task 3: 实现基于轮次的保留策略

**文件**: `game_engine/llm/context_compressor.go`

重写 `CompressHistory()` 中的分割逻辑：

1. 新增 `identifyRounds()` 方法，将消息列表划分为"对话轮次"：
   - 一个轮次定义为：`[user message] -> [assistant + tool_call/result 序列]* -> [final assistant response]`
   - 即从一条 user message 开始，到下一条 user message（不含）为止
2. 保留最近 `RecentKeepRounds` 个完整轮次不被压缩
3. 分割点自动落在轮次边界上，无需再做 `findSafeSplitPoint` 的复杂回溯

```go
type conversationRound struct {
    messages   []Message
```

### Task 4: D&D 领域感知的工具分类压缩

**文件**: `game_engine/llm/context_compressor.go`

改造 `summarizeToolSequence()` 方法，区分两类工具：

- **查询类工具**（get_pc, get_equipment, get_inventory, list_actors, get_race 等）：结果会在下一次 `CollectSummary()` 中重新注入 system prompt，可以**激进压缩**——只保留工具名称，丢弃结果内容
- **动作类工具**（create_pc, add_item, move_actor_to_scene, create_scene, delegate_task 等）：结果不可恢复，需要**保守压缩**——保留关键结果数据（如返回的 entity ID、成功/失败状态）

```go
// 查询类工具前缀列表（结果可从 GameSummary 恢复）
var queryToolPrefixes = []string{"get_", "list_", "query_"}

func isQueryTool(name string) bool {
    for _, prefix := range queryToolPrefixes {
        if strings.HasPrefix(name, prefix) {
            return true
        }
    }
    return false
}
```

压缩策略差异：
- 查询类：`"- 查询[get_pc, get_equipment]: 已获取角色数据"`（不保留结果内容）
- 动作类：`"- 执行[create_pc]: 成功, actor_id=01KPNT8MQG..."`（保留关键 ID 和状态）

### Task 5: Token 估算校准机制

**文件**: `game_engine/llm/context_compressor.go`

新增方法，利用 LLM API 返回的实际 token 用量校准估算：

```go
// CalibrateWithActualUsage 用实际 API 返回的 token 数校准估算模型
func (c *ContextCompressor) CalibrateWithActualUsage(estimatedTokens, actualPromptTokens int) {
    if actualPromptTokens <= 0 || estimatedTokens <= 0 {
        return
    }
    c.lastActualTokens = actualPromptTokens
    c.lastEstimatedTokens = estimatedTokens
    // 指数移动平均更新校准系数
    newRatio := float64(actualPromptTokens) / float64(estimatedTokens)
    c.calibrationRatio = c.calibrationRatio*0.7 + newRatio*0.3
}
```

修改 `EstimateTokens()` 和 `NeedsCompression()` 使用校准系数：

```go
func (c *ContextCompressor) EstimateTokens(messages []Message) int {
    raw := // ... 原有计算
    return int(float64(raw) * c.calibrationRatio)
}
```

### Task 8: 异步后台压缩，不阻塞用户流程

**文件**: `game_engine/llm/context_compressor.go`, `game_engine/react_loop.go`

**核心思路**：当系统检测到需要压缩时，不在当前请求中同步执行压缩，而是启动后台 goroutine 压缩。当前请求继续使用未压缩的完整历史，待后台压缩完成后，下次用户输入时自动替换为压缩后的历史。

**1. ContextCompressor 新增异步状态管理：**

```go
type ContextCompressor struct {
    // ... 已有字段 ...

    // 异步压缩状态
    mu                sync.Mutex
    compressing       bool        // 是否正在后台压缩
    compressedResult  []Message   // 后台压缩完成的结果（待替换）
    compressedReady   bool        // 压缩结果是否就绪
}
```

**2. 新增异步压缩方法：**

```go
// StartAsyncCompress 启动后台异步压缩
// 传入当前历史的快照（copy），在后台执行压缩
// 压缩完成后结果存入 compressedResult，等待下次调用 ApplyCompressedIfReady 时替换
func (c *ContextCompressor) StartAsyncCompress(historySnapshot []Message) {
    c.mu.Lock()
    if c.compressing {
        c.mu.Unlock()
        return // 已有压缩任务在运行，不重复启动
    }
    c.compressing = true
    c.compressedReady = false
    c.mu.Unlock()

    go func() {
        compressed := c.CompressHistory(historySnapshot)

        c.mu.Lock()
        c.compressedResult = compressed
        c.compressedReady = true
        c.compressing = false
        c.mu.Unlock()
    }()
}

// ApplyCompressedIfReady 如果后台压缩已完成，返回压缩后的历史；否则返回 nil
func (c *ContextCompressor) ApplyCompressedIfReady() []Message {
    c.mu.Lock()
    defer c.mu.Unlock()

    if !c.compressedReady {
        return nil
    }

    result := c.compressedResult
    c.compressedResult = nil
    c.compressedReady = false
    return result
}
```

**3. 改造 react_loop.go 中的 `maybeCompressHistory()`：**

将现有的同步压缩改为两步式流程：

```go
func (l *ReActLoop) maybeCompressHistory() {
    if l.compressor == nil {
        return
    }

    // Step 1: 检查是否有已完成的后台压缩结果，如有则应用
    if compressed := l.compressor.ApplyCompressedIfReady(); compressed != nil {
        // 后台压缩基于旧快照，需要将压缩后的摘要 + 快照之后新增的消息合并
        // 简化策略：直接替换（因为压缩保留了最近 N 轮次，且新消息会在下次 observe 时追加）
        beforeLen := len(l.state.History)
        l.state.History = compressed
        log.Info("Applied async compressed history",
            zap.Int("beforeMessages", beforeLen),
            zap.Int("afterMessages", len(compressed)),
        )
        // 同步到 agentContext
        if l.state.agentContext != nil {
            l.state.agentContext.History = l.state.History
        }
        return
    }

    // Step 2: 检查是否需要启动新的后台压缩
    if l.compressor.NeedsCompression(l.state.History) {
        // 拷贝当前历史快照，启动后台压缩
        snapshot := make([]Message, len(l.state.History))
        copy(snapshot, l.state.History)
        l.compressor.StartAsyncCompress(snapshot)
        log.Info("Async compression started in background",
            zap.Int("historyMessages", len(snapshot)),
            zap.Int("estimatedTokens", l.compressor.EstimateTokens(snapshot)),
        )
        // 当前请求继续使用未压缩的完整历史，不阻塞
    }
}
```

**4. 在 `ProcessInput` 入口处也检查压缩结果：**

**文件**: `game_engine/engine.go` — `ProcessInput()` 方法

在添加用户输入到历史之前，先检查是否有已就绪的后台压缩结果：

```go
func (ge *GameEngine) ProcessInput(ctx context.Context, session *GameSession, input string) (string, error) {
    // 在新一轮输入开始时，应用上一轮触发的后台压缩结果
    if ge.reactLoop.compressor != nil {
        if compressed := ge.reactLoop.compressor.ApplyCompressedIfReady(); compressed != nil {
            session.reactLoop.state.History = compressed
        }
    }
    // ... 原有逻辑 ...
}
```

**设计要点**：
- 后台压缩基于历史快照（copy），不影响当前正在进行的对话
- 使用 `sync.Mutex` 保护异步状态，避免竞态
- 同一时间只允许一个后台压缩任务运行（幂等防护）
- 压缩结果在下次 `ProcessInput` 或 `maybeCompressHistory` 时被消费
- 当前实现的纯规则式压缩速度极快（亚毫秒级），异步收益有限；但为未来引入 LLM 摘要压缩（耗时 1-3 秒）预留了架构基础
