# 战斗系统与叙事系统端到端结合

## Summary
- 建立完整闭环：探索/叙事阶段触发遭遇 → `combat_agent` 初始化 dnd-core 战斗并切到 `combat` phase → `GameEngine.ProcessInput` 创建 `CombatSession` 接管 → 战斗结束后生成摘要注入主 ReAct 历史 → phase 回到 `exploration` 后继续由 MainAgent 叙事。
- 以当前代码为准：`CombatDMAgent.RunLoop` 现在主要由 Go 层直接调用 dnd-core，LLM 只负责战斗意图解析和结果叙事，不按旧文档恢复全 tool-calling 内层循环。

## Key Changes
- 在 `game_engine/engine.go` 中补强 phase 交接：
  - 保留现有“请求开始前如果已是 combat 就拦截”的逻辑。
  - 在正常 MainAgent ReActLoop 返回后再次检查 phase；如果本轮刚进入 `combat`，立即创建并初始化 `CombatSession`，把主叙事响应和战斗开场响应合并返回。
  - 抽出一个私有 helper 处理 `CombatResult.CombatEnded`：注入摘要、释放 `session.combatSession`、记录日志，避免三处重复逻辑。
  - `LoadGame` 返回的 `GameSession` 也应设置 `reactLoop: ge.reactLoop`，否则加载存档后战斗摘要注入会潜在 nil panic。
- 给战斗结果补真正的战后摘要：
  - `agent.CombatAgentResult` 增加 `Summary string`。
  - `CombatDMAgent` 维护战斗事件历史，或复用当前 combat history 生成“事实型战斗摘要”。
  - 战斗结束时生成简短中文摘要，包含胜负、关键行动、倒下/存活角色、后续叙事钩子；LLM 不可用时用 dnd-core 状态和已记录事件生成 fallback。
  - `CombatSession.Initialize/ProcessInput` 将 summary 透传到 `CombatResult`。
- 调整主/子 Agent 提示词边界：
  - `main_system.md` 明确：探索阶段遇敌时委托 `combat_agent` 只负责“创建敌人/参战者并开始战斗”，进入 combat 后不要继续解释回合。
  - `combat_system.md` 明确旧 `combat_agent` 在新架构中的职责是战斗启动与非战斗阶段的规则检定；真正回合管理由 `CombatSession/CombatDMAgent` 接管。
  - `narrative_system.md` 增加遭遇交接指令：发现敌人、伏击、主动攻击等场景应描述局势并让 MainAgent 委托 combat 启动，而不是自行结算攻击。
- 增加可离线测试夹具：
  - 新建测试用 `fakeLLMClient`，按 prompt/请求内容返回确定性意图和叙事，避免端到端测试必须依赖真实 OpenAI API。
  - 若 fake 覆盖不足，保留现有真实 LLM 集成测试为慢速/跳过型验证。

## Test Plan
- 新增端到端测试：从 `GameEngine.ProcessInput` 开始，准备 PC、场景、敌人或让 agent 创建敌人，触发战斗；断言 phase 进入 combat 后 `session.combatSession != nil`，响应包含战斗开场或当前回合信息。
- 新增战斗结束恢复测试：用弱敌和 fake LLM 跑到 `CombatEnded=true`，断言 `session.combatSession == nil`，dnd-core phase 回到 `exploration`，主 ReAct 历史包含 `[战斗摘要]`。
- 新增继续叙事测试：战斗结束后再调用一次 `ProcessInput("我检查战场")`，断言走 MainAgent/ReActLoop 正常路径，不再进入 CombatSession。
- 保留真实 LLM 集成测试 `TestCombatSessionFullBattle`，但不把它作为快速 CI 的唯一验收。
- 验证命令：`go test ./game_engine/...` 和 `go test ./...`；当前环境里 `go` 命令不可用，实施时需在有 Go toolchain 的环境运行。

## Assumptions
- 第一阶段不改 dnd-core API，只使用当前已有的 `SetupCombat`、`NextTurnWithActions`、`ExecuteTurnAction`、`GetAvailableActions` 等接口。
- 战斗奖励、掉落、任务推进先不自动结算；战后摘要只把事实交还 MainAgent，由后续叙事/库存/任务 agent 处理战利品与剧情推进。
- 旧 `combat_agent` 暂时保留，用于探索阶段启动战斗和非完整战斗 session 的规则相关任务；不在本次计划中删除或大重构。
